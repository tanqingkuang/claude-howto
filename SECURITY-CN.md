# 安全政策

## 概览

Claude How To 项目的安全性对我们非常重要。本文档概述了我们的安全实践，并说明如何负责任地报告安全漏洞。

## 支持的版本

我们为以下版本提供安全更新：

| 版本 | 状态 | 支持期限 |
|------|------|----------|
| 最新（main） | ✅ 活跃 | 当前版本 + 6 个月 |
| 1.x releases | ✅ 活跃 | 直到下一个主版本发布 |

**注意**：作为一个教育型指南项目，我们更关注维护当前最佳实践和文档安全，而不是传统的软件版本支持。更新会直接应用到 main 分支。

## 安全实践

### 代码安全

1. **依赖管理**
   - 所有 Python 依赖都固定在 `requirements.txt` 中
   - 通过 Dependabot 和人工审查定期更新
   - 每次提交都通过 Bandit 做安全扫描
   - 使用 pre-commit hooks 进行安全检查

2. **代码质量**
   - 使用 Ruff 做 lint，捕获常见问题
   - 使用 mypy 做类型检查，防止类型相关漏洞
   - 通过 pre-commit hooks 强制执行标准
   - 所有变更在合并前都会经过审查

3. **访问控制**
   - `main` 分支启用分支保护
   - 合并前必须有审查通过
   - 合并前必须通过状态检查
   - 仓库写入权限受限

### 文档安全

1. **示例中不放密钥**
   - 所有示例里的 API key 都是占位符
   - 从不硬编码凭据
   - `.env.example` 文件会展示所需变量
   - 明确提示如何管理密钥

2. **安全最佳实践**
   - 示例展示安全模式
   - 文档中高亮安全警告
   - 链接到官方安全指南
   - 在相关章节讨论凭据处理

3. **内容审查**
   - 所有文档都会审查安全问题
   - 贡献指南中包含安全注意事项
   - 验证外部链接和引用

### 依赖安全

1. **扫描**
   - Bandit 扫描所有 Python 代码中的漏洞
   - 通过 GitHub 安全告警检查依赖漏洞
   - 定期进行人工安全审计

2. **更新**
   - 安全补丁会尽快应用
   - 主版本会谨慎评估
   - 变更日志会记录与安全相关的更新

3. **透明度**
   - 安全更新会记录在提交中
   - 漏洞披露会以负责任的方式处理
   - 在适当情况下发布公开安全通告

## 报告漏洞

### 我们关心的安全问题

我们欢迎以下类型的报告：
- **代码漏洞**：脚本或示例中的漏洞
- **依赖漏洞**：Python 包中的漏洞
- **密码学问题**：任何代码示例中的加密弱点
- **认证/授权缺陷**：文档中的认证授权问题
- **数据泄露风险**：配置示例里的暴露风险
- **注入漏洞**：SQL、命令等注入
- **SSRF/XXE/路径穿越**：相关安全问题

### 不在范围内的安全问题

以下内容不属于本项目的范围：
- Claude Code 本身的漏洞（请向 Anthropic 报告）
- 外部服务或库的问题（请向上游项目报告）
- 社会工程学或用户教育问题（不适用于本指南）
- 没有概念验证（PoC）的理论漏洞
- 已通过官方渠道报告的依赖漏洞

## 如何报告

### 私密报告（推荐）

**对于敏感安全问题，请使用 GitHub 的私密漏洞报告：**

1. 访问：https://github.com/luongnv89/claude-howto/security/advisories
2. 点击 “Report a vulnerability”
3. 填写漏洞详情
4. 包含以下内容：
   - 漏洞的清晰描述
   - 受影响的组件（文件、章节、示例）
   - 潜在影响
   - 复现步骤（如果适用）
   - 建议修复方案（如果你有）

**接下来会发生什么：**
- 我们会在 48 小时内确认收到报告
- 我们会调查并评估严重性
- 我们会与你协作制定修复方案
- 我们会协调披露时间线
- 我们会在安全通告中致谢你（除非你希望匿名）

### 公开报告

对于非敏感问题或已经公开的问题：

1. **创建 GitHub Issue**，并加上 `security` 标签
2. 在 issue 中包含：
   - 标题：`[SECURITY]` 加上简短描述
   - 详细说明
   - 受影响的文件或章节
   - 潜在影响
   - 建议修复方案

## 漏洞响应流程

### 评估（24 小时内）

1. 我们确认收到报告
2. 使用 [CVSS v3.1](https://www.first.org/cvss/v3.1/specification-document) 评估严重性
3. 判断是否在范围内
4. 向你发送初步评估

### 开发（1-7 天）

1. 我们开发修复方案
2. 我们审查并测试修复
3. 我们创建安全通告
4. 我们准备发布说明

### 披露（取决于严重性）

**Critical（CVSS 9.0-10.0）**
- 立即发布修复
- 发布公开通告
- 提前 24 小时通知报告者

**High（CVSS 7.0-8.9）**
- 48-72 小时内发布修复
- 提前 5 天通知报告者
- 在发布时公开通告

**Medium（CVSS 4.0-6.9）**
- 在下一次常规更新中发布修复
- 在发布时公开通告

**Low（CVSS 0.1-3.9）**
- 在下一次常规更新中包含修复
- 在发布时附带通告

### 发布

我们发布的安全通告会包含：
- 漏洞描述
- 受影响的组件
- 严重性评估（CVSS 分数）
- 修复版本
- 规避方法（如果适用）
- 报告者致谢（经允许）

## 报告者最佳实践

### 报告前

- **验证问题**：你能稳定复现吗？
- **搜索现有 issue**：是否已经有人报告过？
- **检查文档**：是否已有安全使用说明？
- **测试修复**：你建议的修复是否有效？

### 报告时

- **具体明确**：提供准确的文件路径和行号
- **补充上下文**：为什么这会是安全问题？
- **展示影响**：攻击者可能做什么？
- **提供步骤**：我们如何复现？
- **建议修复**：你会怎么修？

### 报告后

- **保持耐心**：我们的资源有限
- **及时响应**：尽快回答后续问题
- **保持保密**：在修复前不要公开披露
- **尊重协调**：按我们的披露时间线进行

## 安全头部与配置

### 仓库安全

- **分支保护**：main 分支的变更需要 2 个批准
- **状态检查**：所有 CI/CD 检查都必须通过
- **CODEOWNERS**：关键文件指定审查人
- **签名提交**：建议贡献者使用

### 开发安全

```bash
# 安装 pre-commit hooks
pre-commit install

# 在本地运行安全扫描
bandit -c pyproject.toml -r scripts/
mypy scripts/ --ignore-missing-imports
ruff check scripts/
```

### 依赖安全

```bash
# 检查已知漏洞
pip install safety
safety check

# 或使用 pip-audit
pip install pip-audit
pip-audit
```

## 贡献者安全指南

### 编写示例时

1. **绝不要硬编码密钥**
   ```python
   # ❌ 不好
   api_key = "sk-1234567890"

   # ✅ 更好
   api_key = os.getenv("API_KEY")
   ```

2. **提示安全影响**
   ```markdown
   ⚠️ **安全提示**：切勿将 `.env` 文件提交到 git。
   请立即将其加入 `.gitignore`。
   ```

3. **使用安全默认值**
   - 默认启用认证
   - 在适用时使用 HTTPS
   - 校验并清理输入
   - 使用参数化查询

4. **记录安全注意事项**
   - 解释为什么安全很重要
   - 展示安全与不安全的模式
   - 链接到权威来源
   - 以显著方式显示警告

### 审查贡献时

1. **检查是否暴露密钥**
   - 扫描常见模式（api_key=、password=）
   - 审查配置文件
   - 检查环境变量

2. **验证安全编码实践**
   - 没有硬编码凭据
   - 输入校验正确
   - 认证/授权安全
   - 文件处理安全

3. **测试安全影响**
   - 是否可能被滥用？
   - 最坏情况是什么？
   - 是否存在边界情况？
